# 🧩 Правила модульной архитектуры

Проект использует модульную архитектуру: каждая функциональная область оформлена как независимый модуль со своей структурой, логикой и интерфейсом взаимодействия.

**Цель** — масштабируемость, читаемость и переиспользуемость кода.

---

## 📁 Расположение

- **Модули** — директория `modules/`
- **Общее и переиспользуемое** (UI, утилиты, хуки, типы) — директория `shared/`

> **В этом проекте:** пока `shared/` не введён, роль общего слоя выполняют `lib/`, `hooks/` и `components/ui/`. Импорты из них считаются разрешёнными из любого модуля и из `app/`.

---

## 🧱 Структура модуля

Каждый модуль имеет структуру:

```
modules/
  <module-name>/
    api/           # запросы к backend
    components/    # UI модуля
    hooks/         # хуки только этого модуля
    types/         # типы, интерфейсы, enum
    store/         # (опционально) Zustand, Redux, Context
    utils/         # (опционально) вспомогательные функции модуля
    index.ts       # barrel-экспорт публичного API
```

| Папка         | Назначение |
|---------------|------------|
| `components/` | UI-компоненты модуля |
| `hooks/`      | React-хуки только этого модуля |
| `api/`        | Запросы и взаимодействие с backend |
| `types/`      | Типы, интерфейсы, перечисления |
| `store/`      | *(опционально)* Локальное состояние |
| `utils/`      | *(опционально)* Вспомогательные функции модуля |
| `index.ts`    | Barrel-экспорт публичных сущностей |

---

## 🔗 Зависимости и импорты

- Модули **не зависят друг от друга напрямую**.
- Взаимодействие — только через **публичный API** (`index.ts`) или через **`shared/`** (или `lib/`, `hooks/`, `components/ui/` в текущей структуре).
- **Обязательны абсолютные импорты:**

```ts
import { LoginForm } from '@/modules/auth'
import { Button } from '@/shared/ui'
```

- **Запрещено** импортировать внутренности другого модуля:

```ts
// ❌ Неправильно
import { login } from '@/modules/user/api'
import { UserCard } from '@/modules/user/components/UserCard'
```

## 🧩 Именование

| Тип                | Стиль              | Пример |
|--------------------|--------------------|--------|
| Файлы и папки      | kebab-case         | `auth-form.tsx`, `use-auth.ts` |
| Компоненты         | PascalCase         | `LoginForm.tsx` |
| Хуки               | camelCase, префикс `use` | `useAuth.ts` |
| Типы / интерфейсы  | PascalCase (опц. `I`-префикс) | `User`, `IAuthResponse` |
| Store              | имя модуля + Store | `authStore.ts` |

## 🧠 Экспорт и принципы

- У каждого модуля **обязателен** `index.ts` — единая точка публичных экспортов.
- Модуль следует **SRP** (Single Responsibility Principle).
- Роутинг, layout, тема, глобальный конфиг — **вне** `modules/`.
- Общие стили, компоненты и хуки — в **`shared/`**.

**Пример `index.ts`:**

```ts
export * from './components'
export * from './hooks'
export * from './api'
export * from './types'
```

## ✅ Тесты и качество

- Тесты (unit, integration) — в `modules/<name>/__tests__/`.
- ESLint, Prettier и Husky — обязательны.
- Каждый PR проходит линтинг и форматирование.
- Покрытие тестами — минимум 50% для критичных модулей (auth, billing, core). Для лендинга допустимо ниже — по договорённости.

## 💡 Лучшие практики

### Архитектура

- **Разделяй бизнес-логику и UI:** в компонентах не должно быть запросов к API; логика — в хуках или store.
- Предпочитай чистые функции и композицию, не наследование.
- Новая фича — новый модуль; без необходимости не расширяй чужие модули.
- Публичные функции и компоненты — с явной типизацией.
- Минимум связей между модулями; общее — только через `shared/`.

### Качество кода

- Нет «магических значений» — константы в `constants.ts`.
- Вложенность JSX — не более 3 уровней.
- Компонент — не более 200 строк.
- Модуль самодостаточен: его можно удалить — проект собирается.

### Команда

- Один PR — одна задача.
- Комментарии объясняют **почему**, а не что делает код.
- TODO/FIXME — с автором и датой.
- Новые модули проходят код-ревью.

## 🔍 Примеры импортов

```ts
// ✅ Правильно
import { useAuth } from '@/modules/auth'
import { Button } from '@/shared/ui'

// ❌ Неправильно
import { login } from '@/modules/user/api'
import { something } from '@/modules/other-module/utils/helpers'
```

## 🧭 Итог

Модульная архитектура даёт:

- изоляцию кода;
- простоту тестирования;
- независимость при доработках;
- возможность параллельной работы разработчиков.
